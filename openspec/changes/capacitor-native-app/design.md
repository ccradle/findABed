## Context

FABT is a React 19 + Vite + TypeScript PWA serving shelter coordinators (desktop) and outreach workers (mobile). The codebase analysis shows 85% native-readiness: fetch-based API layer, 44px touch targets, mobile-responsive layout, offline queue, dark mode, and i18n. The main gaps are: localStorage for tokens, `window.location.href` for routing, missing icon assets, and no iOS meta tags.

Capacitor was selected over React Native (would require UI rewrite) and TWA (Android-only, no native APIs). Capacitor wraps the existing web build in native WebView containers with plugin access to device APIs.

## Goals / Non-Goals

**Goals:**
- Darius receives push notifications for bed availability even when the app is backgrounded
- Dr. Whitfield's hospital can pre-install via MDM without browser dependency
- Offline hold queue survives iOS 7-day cache eviction (SQLite replaces IndexedDB)
- Token storage uses hardware-backed keychain (Face ID / Touch ID protected)
- Published on both Apple App Store and Google Play Store
- OTA updates for JS-only changes (bypasses store review for critical fixes)
- PWA continues to work unchanged for web users

**Non-Goals:**
- Rewriting UI components in React Native (Capacitor renders existing React DOM)
- Replacing SSE with push for foreground users (SSE is better for real-time; push is for background)
- Building a separate admin/coordinator native app (they stay on the PWA at the desk)
- Geolocation-based shelter search (future feature, not in this change)

### Token storage migration

**Current:** `localStorage.getItem('fabt_access_token')`
**New:** Platform-aware storage abstraction:

```typescript
// services/storage.ts
import { Capacitor } from '@capacitor/core';
import { SecureStorage } from '@capacitor-community/secure-storage';

export const tokenStorage = {
  async get(key: string): Promise<string | null> {
    if (Capacitor.isNativePlatform()) {
      const result = await SecureStorage.get({ key });
      return result.value;
    }
    return localStorage.getItem(key);
  },
  async set(key: string, value: string): Promise<void> {
    if (Capacitor.isNativePlatform()) {
      await SecureStorage.set({ key, value });
    } else {
      localStorage.setItem(key, value);
    }
  },
  async remove(key: string): Promise<void> {
    if (Capacitor.isNativePlatform()) {
      await SecureStorage.remove({ key });
    } else {
      localStorage.removeItem(key);
    }
  }
};
```

This abstraction keeps the PWA working with localStorage while native apps use hardware-backed keychain. The `AuthContext`, `api.ts`, and `SessionTimeoutWarning` migrate to use `tokenStorage` instead of direct `localStorage` calls.

**Critical: in-memory token cache.** SecureStorage reads are async, but the API request interceptor needs the token synchronously. Solution: on login, write to SecureStorage AND cache in a module-level variable. On each request, read from memory (synchronous). On logout, clear both. On app launch (native), read from SecureStorage into memory before rendering the first route. This avoids making the entire fetch pipeline async.

### Push notification architecture

```
Event occurs (availability update, referral response)
    │
    ├── NotificationService.broadcast() → SSE stream (web PWA, foreground)
    │
    └── PushNotificationSender.send() → FCM HTTP v1 API
                                            │
                                            ├── Android: FCM → device
                                            └── iOS: FCM → APNs → device
```

**Backend:**
- New `PushNotificationSender` service using direct FCM HTTP v1 API calls (no `firebase-admin` SDK — keeps backend cloud-provider-agnostic, uses `java.net.http.HttpClient` already in the project)
- New `DeviceTokenRepository` (Flyway migration: `device_token` table with `user_id`, `token`, `platform`, `created_at`)
- New endpoint: `POST /api/v1/devices` (register FCM token), `DELETE /api/v1/devices/{token}` (unregister)
- `NotificationService.broadcast()` calls both SSE dispatch and FCM push

**Frontend (native only):**
- On app launch, request push permission, get FCM token, register via `POST /api/v1/devices`
- On push received while foregrounded: show in-app notification (same as SSE event)
- On push received while backgrounded: OS shows native notification banner
- Tapping the notification deep-links to the relevant screen (bed search, referral detail)

### Offline storage: SQLite for hold queue

Replace `offlineQueue.ts` IndexedDB with SQLite on native platforms:

```typescript
import { Capacitor } from '@capacitor/core';
import { CapacitorSQLite } from '@capacitor-community/sqlite';

// On native: use SQLite (persistent, no eviction)
// On web: keep IndexedDB (existing behavior)
if (Capacitor.isNativePlatform()) {
  // SQLite-backed queue
} else {
  // Existing IndexedDB queue (offlineQueue.ts)
}
```

This solves the iOS 7-day eviction problem for native users while keeping web users on the existing IndexedDB path.

### Biometric authentication

Optional enhancement: after initial password login, offer to enable biometric unlock. On subsequent launches, the user taps Face ID / Touch ID instead of entering credentials. The JWT token stored in SecureStorage is unlocked by the biometric check.

### Routing abstraction

Replace `window.location.href = '/login'` in api.ts with a navigation abstraction:

```typescript
// In api.ts, on 401:
if (Capacitor.isNativePlatform()) {
  // Use React Router programmatic navigation
  window.dispatchEvent(new CustomEvent('fabt:auth-expired'));
} else {
  window.location.href = '/login';
}
```

The `AuthContext` listens for `fabt:auth-expired` and uses `navigate('/login')`.

### Apple App Store Guideline 4.2 compliance

To avoid rejection as a "repackaged website," the native app must demonstrate native value:
1. **Push notifications** — native APNs integration (not Web Push)
2. **Biometric auth** — Face ID / Touch ID for login
3. **SQLite offline** — persistent local database
4. **Deep linking** — `fabt://shelter/{id}` opens specific shelter
5. **Privacy manifest** — `PrivacyInfo.xcprivacy` declaring data collection practices

## Risks

- **Apple rejection risk:** Mitigated by 5 native features listed above. If initial submission is rejected, iterate with Apple reviewer feedback.
- **Firebase dependency:** FCM is free tier, but adds a Google Cloud dependency. Alternative: direct APNs for iOS + FCM for Android. For simplicity, FCM handles both (it proxies to APNs automatically).
- **Capacitor version upgrades:** Pin to a specific major version. Capacitor 6.x is current stable.
- **Dual storage paths:** SQLite (native) + IndexedDB (web) for the offline queue. The abstraction layer keeps this clean but adds a code path to test.
- **MDM distribution:** Requires Apple Developer Enterprise Program ($299/year) for in-house distribution, or standard App Store with MDM-managed installation. TestFlight supports up to 10,000 external testers as an alternative.
